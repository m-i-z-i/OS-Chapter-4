§ 4.0 Core Concept

What is a Process?

A process is simply a running program. This one-sentence definition is the foundation of everything in this chapter.

Every single app on your computer — your browser, music player, game — exists first as a program: a file sitting on your hard drive containing instructions. That file does absolutely nothing on its own. It's passive, static, lifeless bytes. The moment you run it, the Operating System wakes it up and transforms it into a process — a living, active entity that uses the CPU, consumes memory, and does actual work.

Program — Passive

The Recipe

  • Lifeless static bytes stored on disk
  • Does absolutely nothing on its own
  • Can be copied, deleted, shared
  • One program file exists once on disk
  • Like a recipe sitting in a cookbook
Process — Active

The Cooking

  • A running entity with dynamic state
  • Actively executing instructions on CPU
  • Uses real memory, real CPU time
  • One program can spawn many processes
  • Like actually cooking from that recipe
Definition

A process is simply a running program. It is the OS's most fundamental abstraction — the illusion that each program has the machine to itself.

Real-world analogy

A program is like a music score — sheets of paper with notes on them. A process is the orchestra actually performing that score. The score sits still; the performance is alive, dynamic, and real-time. You can have multiple orchestras perform the same score simultaneously — just like you can open multiple browser windows from one program.

This distinction matters because the OS manages processes, not programs. When you press Ctrl+Alt+Del and see the Task Manager, every row is a process — not a program file. Chrome might show 10 rows because it runs one process per tab for safety and performance.

§ 4.1 The Core Problem

CPU Virtualization — The Illusion of Many CPUs

How can 200 programs run "simultaneously" on a machine with only 8 CPU cores? The answer is one of the most elegant tricks in computer science.

Your computer physically has a limited number of CPU cores — maybe 4, 8, or 16. Yet right now you might have a browser, a music player, a code editor, an antivirus, and dozens of background services all "running" at once. How?

The OS creates an illusion. It switches between processes so rapidly — thousands of times per second — that it feels like everything is happening simultaneously. This technique is called time sharing, and it is the fundamental mechanism behind CPU virtualization.

Analogy — the busy chef

Imagine a chef at a busy restaurant. They don't cook one dish start-to-finish before starting the next. Instead: start the pasta boiling, go chop vegetables, flip the steak, stir the sauce, check the pasta, and so on. To the diners, all dishes are being prepared "at once" — but the chef is just switching tasks very quickly. The OS is that chef. Processes are the dishes.

Time Sharing — CPU

How the CPU is shared

The OS runs process A for a tiny time slice (e.g. 10 milliseconds), saves its state, then runs process B for 10ms, then C... When it comes back to A, it restores its state and continues. Each process thinks it has the whole CPU. Cost: a small performance overhead on each switch.

Space Sharing — Disk

How disk is shared differently

Disk space is divided physically. Each file owns its sectors exclusively — there is no switching or sharing. File A's blocks belong to A until deleted. This is fundamentally different from time sharing — it is about dividing space, not time.

Key insight

Virtualizing the CPU means making each process believe it has a dedicated CPU, when in reality it shares one physical CPU with many others. This is the OS's greatest magic trick.

The challenge is doing this efficiently and fairly. The OS must decide: which process runs next? For how long? What happens when a process is waiting for disk I/O? These are questions of scheduling policy, which we'll cover in Chapter 7. For now, Chapter 4 just introduces the abstraction.

§ 4.1 OS Design Philosophy

Mechanisms vs Policies

One of the most important design principles in operating systems: separate what you can do from what you decide to do.

OS designers discovered early on that mixing implementation details with decision-making logic creates fragile, hard-to-change systems. The solution is to rigorously separate two things:

Mechanism — "The HOW"

Low-level machinery

A mechanism is the protocol or procedure for doing something. It knows how to perform an action, but makes no decisions about when or why.


Example: The context switch is a mechanism. It knows the exact steps to save process A's CPU registers to memory and load process B's registers from memory. It doesn't decide whether to switch — it just knows how.

Policy — "The WHICH"

High-level intelligence

A policy is the algorithm or strategy for making decisions. It decides which process runs and when, using mechanisms to enforce those decisions.


Example: The scheduling policy is a policy. It decides: run the shortest job next? Run the highest-priority process? Round-robin between all? It uses the context switch mechanism to implement its choice.

Analogy — traffic system

The traffic light is a mechanism — it knows how to turn red, yellow, and green. The timing controller is the policy — it decides how long each phase lasts based on traffic density. You can upgrade the controller (smarter timing algorithm) without replacing the lights. You can install different lights without rewriting the controller logic. They're independent.

Design Tip

Modular separation of mechanisms and policies allows policies to change without rewriting low-level mechanisms. This is why OS researchers can experiment with dozens of scheduling algorithms without touching the context switch code.

§ 4.1 What a Process Contains

The Machine State — What's Inside a Process?

To understand what the OS must save and restore when switching between processes, you need to know exactly what a process "owns" at any given moment.

A process is more than just running code. It has three fundamental components — think of these as the process's "possessions." To pause a process and resume it later perfectly (as if it never stopped), the OS must save all three completely and restore them exactly.

1 · Memory

Address Space

The chunk of RAM that belongs exclusively to this process. Contains its code (machine instructions), global variables, the stack, and the heap. No other process can read or write this memory — it is private and isolated.

2 · CPU State

Registers

The CPU's current working state for this process. The most important are:

Program Counter (PC): Which instruction to execute next — like a bookmark in the code.

Stack Pointer (SP): Points to the top of the call stack.

Plus many general-purpose registers holding temporary values.

3 · I/O State

I/O Information

A list of open file descriptors — the channels to the outside world that this process has open:

fd 0: stdin (keyboard input)
fd 1: stdout (screen output)
fd 2: stderr (error output)
fd 3+: any files/sockets the program opened

Analogy — pausing a video game

When you save a video game and come back later, the game must restore: your position (address space), what buttons you were pressing (registers), and which inventory items were open (I/O state). If it only saves your position but forgets your health and inventory, you can't resume seamlessly. The OS does exactly this — a perfect save and restore of all three components — on every context switch.

Memory Layout Visualization

Why this matters

To pause process A and resume it later, the OS saves its registers (including PC and SP) into the PCB. The address space stays in memory (or is saved to disk). The file descriptors are tracked in the kernel. When A resumes, everything is restored — it has no idea it was ever paused.

§ 4.2 Process API

The Process API — 5 Things Every OS Must Provide

Every operating system — Windows, Linux, macOS — must provide these five fundamental operations for managing processes.

An API (Application Programming Interface) is a set of functions that programs can call to ask the OS to do things. For processes, the OS must provide at minimum:

1

Create — Launch a new process

The OS must be able to create new processes. When you double-click an app, the OS runs a "create process" call. In Unix/Linux, this is done via fork() + exec(). The shell uses this constantly — every command you type creates a new process.

2

Destroy — Forcefully terminate a process

The OS must be able to kill a process that refuses to stop on its own — a crashed app, an infinite loop, or malicious code. In Unix: kill(). In Windows: TerminateProcess(). This is the "force quit" of the OS world. Most processes exit on their own via exit(), but having a forced destroy option is essential.

3

Wait — Pause until an event completes

A parent process often needs to wait for its child process to finish before proceeding. In Unix: wait() or waitpid(). Example: a shell running a command waits for that command to finish before showing the next prompt. This is also how the parent cleans up zombie processes (more on this later).

4

Control — Suspend and resume

The OS must be able to temporarily pause a process (suspend) and later continue it (resume). In Unix, signals like SIGSTOP and SIGCONT do this. Ctrl+Z in a terminal suspends a process — it moves from Running to a stopped state without being killed.

5

Status — Query information about a process

Get information about a process's state: is it running, sleeping, stopped? How much CPU has it used? What's its PID? In Unix: ps, top, getpid(). This is what Task Manager and Activity Monitor use to show you the process list.

Design tip

The book notes that a good OS provides these APIs as a clean, simple interface — but the implementation underneath (the mechanism) can be very complex. The user/programmer should never need to know those details.

§ 4.3 How Processes Are Born

Process Creation — From Disk to Running

What actually happens between you double-clicking an app and the app appearing on screen? The OS follows a precise sequence of steps every single time.

Creating a process is not instantaneous. The OS must carefully set up everything the process needs before it can run a single instruction of your code. Here are the steps in order:

1

Load code and data from disk into memory

The OS reads the executable file (e.g. .exe on Windows, ELF binary on Linux) from the hard drive and copies its bytes into the process's address space in RAM. This is essential — the CPU can only execute instructions from RAM, not directly from disk.


Eager loading (old way): Load all code and data upfront before starting. Simple but slow for large programs.

Lazy loading (modern way): Load pages from disk only when the process actually tries to access them. This is called demand paging and makes programs start much faster — you only pay for what you use.

2

Allocate the runtime stack

The OS creates the process's stack — a fixed-size memory region for function calls. The stack holds: local variables (e.g. int x = 5 inside a function), function parameters, and return addresses (where to go back to after a function finishes).


The OS also initializes the stack with the program's command-line arguments (argc and argv). When you type ls -la /home, the shell creates a process with argc=3 and argv=["ls", "-la", "/home"] on the stack.

3

Allocate the heap

The OS sets up the process's heap — a memory region for dynamic allocation. Unlike the stack (which has a fixed size), the heap can grow and shrink at runtime. When your program calls malloc() in C or new in C++/Java, it requests memory from the heap.


The heap starts small (often just a few pages) and expands as the program requests more. The OS provides the brk() / sbrk() system call to grow it.

4

Set up I/O — open standard file descriptors

Before your code runs a single line, the OS opens three file descriptors automatically:


fd 0 — stdin — Standard input. Connected to the keyboard (or a pipe if you redirect input). Your program reads user input from here.

fd 1 — stdout — Standard output. Connected to the terminal (or a file if you redirect). printf() and cout write here.

fd 2 — stderr — Standard error. Also connected to the terminal. Error messages go here — this way you can redirect stdout to a file without losing error messages.


This is why you can immediately use printf() without opening any files — the OS already did it for you.

5

Jump to main() — the process starts running

With everything set up, the OS transfers control to the process's entry point: the main() function. The CPU's Program Counter is set to the address of main(), and the process begins executing.


At this point, the OS steps back. Your code is running. The OS still watches over it (handling system calls, page faults, etc.) but the process is now alive and in control of its own execution.

Exam note

You may be asked about eager vs lazy loading. Eager = load everything before starting (simple, but wastes time for large programs). Lazy = load pages on demand as needed (fast start, used by all modern OSes via demand paging). Modern OSes always use lazy loading.

§ 4.4 Process Lifecycle

Process States — Where is a Process at Any Moment?

A process is not always running. At any given moment, it is in one of several states. Understanding these states is the key to understanding how the OS manages processes.

The Three Core States

State 1

Running

The process is actively executing instructions on the CPU right now. The CPU's Program Counter points to this process's code. Only one process per CPU core can be in the Running state at a time. This is the only state where actual work gets done.

State 2

Ready

The process is ready and willing to run but the OS has not yet scheduled it. It's waiting in a ready queue for its turn on the CPU. It could run right now if given the CPU — there's just nothing stopping it except the scheduler hasn't chosen it yet.

State 3

Blocked

The process is waiting for an external event — typically I/O — to complete before it can proceed. A blocked process cannot make progress even if given the CPU, because it's waiting for something outside the CPU (e.g., a disk read, network packet, keyboard input).

Analogy — airport security line

Running = you're at the scanner being processed. Ready = you're standing in the queue, fully prepared, just waiting your turn. Blocked = you stepped aside because you forgot to take off your shoes — you're not ready to be processed yet, so you're not in the queue. Once you're ready, you rejoin the queue (Ready state).

Critical exam rule

A process executes instructions ONLY when it is in the RUNNING state. Being "Ready" means it could run — not that it is running. Being "Blocked" means it cannot run even if the CPU is free. Never confuse Ready with Running.

Key Takeaway

The OS maximizes CPU utilization by switching to another process whenever one blocks for I/O. Rather than letting the CPU sit idle waiting for a slow disk, it immediately puts that CPU to productive use. Without this, a single disk read would waste the entire machine.

Notice how P₀ resumed from exactly where it left off at time 9. That's the magic of saving and restoring machine state — P₀ never knew it was paused. From P₀'s perspective, the disk read just took a moment, when in reality three other things happened during that time.

§ 4.5 Data Structures

The Process Control Block (PCB)

The OS tracks every single process using a data structure called the Process Control Block. This is the OS's "dossier" on each process — everything it needs to know to manage it.

For the OS to switch between processes (context switch), pause them, resume them, and track their state, it needs to store information about each process somewhere. That somewhere is the PCB — one PCB per process, stored in the OS kernel.

In the textbook (xv6, a teaching OS), the PCB is defined as:

proc.h — Process Control Block (PCB)
struct proc {
    /* State this process is in */
    enum proc_state  state;    // RUNNING, READY, BLOCKED, ZOMBIE, etc.

    /* Saved CPU registers — stored here during a context switch */
    struct context   context;  // Contains PC, SP, and all general registers

    /* Memory */
    char             *mem;     // Pointer to start of process address space

    /* Process relationships */
    struct proc     *parent;  // Pointer to the parent process (who created this)

    /* Identifier */
    int              pid;      // Process ID — unique number for this process
};

What Each Field Does

state

Current state

Which state is this process in right now? RUNNING, READY, BLOCKED, or ZOMBIE. The OS updates this field constantly as the process moves between states.

context

Saved registers

When the OS does a context switch (pauses this process), it dumps all the CPU registers here. When this process runs again, it loads them back. This is how the process resumes seamlessly from where it left off.

mem

Memory pointer

A pointer to the start of this process's address space. The OS uses this to know where in physical RAM this process's memory lives.

parent

Parent process

Every process (except the first — init) has a parent. This pointer tracks who created this process. The parent-child relationship is important for wait() and zombie cleanup.

pid

Process ID

A unique integer assigned to each process. The first process (init) gets PID 1. Every new process gets the next available number. ps and Task Manager display PIDs. You use them with kill <pid>.

Context switch detail

During a context switch from process A to B: (1) Save A's CPU registers into A's PCB.context. (2) Update A's PCB.state to READY. (3) Load B's registers from B's PCB.context into the CPU. (4) Update B's PCB.state to RUNNING. The CPU now continues from B's saved Program Counter — B picks up exactly where it left off.

★ EXAM Critical Concept

The Zombie State — Dead but Not Gone

One of the most frequently tested concepts in this chapter: what happens to a process after it calls exit()? The answer is more interesting than "it disappears."

When a process calls exit() — either explicitly or by returning from main() — you might think it simply disappears. In reality, it enters a special state called ZOMBIE.

Definition — Zombie State

A process that has finished executing (called exit()) but whose PCB has not yet been cleaned up by its parent process. It is dead, but its entry still exists in the process table.

Why Does the Zombie State Exist?

When a child process exits, it may have important information the parent needs:

1

Child calls exit(status_code)

The child process finishes and calls exit(0) (success) or exit(1) (failure). The OS needs to preserve this exit status code somewhere so the parent can read it later. That "somewhere" is the PCB — which is why the PCB can't be deleted yet.

2

Process enters ZOMBIE state

The OS frees most of the process's resources — its address space (RAM), open file descriptors, etc. — but keeps the PCB alive. The process is "mostly dead." It cannot run, it has no memory, but it still has an entry in the process table with its PID and exit status.

3

Parent calls wait() to collect the exit status

When the parent process calls wait() or waitpid(), the OS gives the parent the child's exit status. The OS then completely removes the child's PCB. The process is now fully gone — no longer a zombie.

!

Problem: Parent never calls wait()

If the parent process never calls wait(), the zombie lingers forever (until the parent itself dies). A zombie consumes an entry in the process table. If thousands of zombies accumulate, the process table fills up and the system can't create new processes — a denial-of-service attack or serious bug.

Analogy

Think of a zombie process like an employee who quit their job but whose name is still on the payroll database. The company (OS) keeps their record so management (the parent) can do their exit interview and final paperwork (read the exit status). Once the exit interview is done (parent calls wait()), the record is deleted. If management never does the exit interview, the record stays forever — wasting database space.

Exam definition to memorize

Zombie state: A process that has called exit() and finished executing, but whose PCB entry has not yet been removed by its parent process. The parent must call wait() to collect the exit status, after which the OS removes the PCB completely. A zombie holds no memory or file descriptors — only a PCB entry with PID and exit status.

The 10 Things You Must Know

1

Process = running program

A program is passive bytes on disk. A process is an active, running instance. The OS transforms one into the other. One program can spawn multiple processes.

2

CPU virtualization via time sharing

The OS rapidly switches between processes (time slicing) to create the illusion that all programs run simultaneously. Each gets a small time slice. Cost: performance overhead on each switch.

3

Mechanism (HOW) vs Policy (WHICH)

Context switch = mechanism (how to switch). Scheduling algorithm = policy (who runs next). Keep them separate so you can change the policy without rewriting the mechanism.

4

Machine state: 3 components

Every process has: (1) Address space (private memory with code, data, stack, heap), (2) Registers (PC, SP, general regs), (3) I/O info (open file descriptors). All must be saved/restored on context switch.

5

Process creation order

1→ Load code/data. 2→ Allocate stack (+ set argc/argv). 3→ Allocate heap. 4→ Set up stdin/stdout/stderr. 5→ Jump to main(). Modern OSes load lazily (demand paging) not eagerly.

6

3 states: Running, Ready, Blocked

Running = actively on CPU. Ready = wants CPU, waiting in queue. Blocked = waiting for I/O, can't run. Process executes instructions ONLY in the Running state.

7

OS maximizes CPU by using I/O blocking

When process A blocks for I/O, the OS immediately runs process B. This keeps the CPU busy instead of idle. When A's I/O finishes, A moves back to Ready, eventually Running again.

8

PCB — Process Control Block

The OS's data structure for each process. Contains: state, context (saved registers), mem (address space pointer), parent pointer, pid. Registers are saved here during every context switch.

★

Zombie state (exam favourite!)

When a process calls exit(), it enters ZOMBIE state. It's done running but its PCB stays until the parent calls wait(). The PCB holds the exit status code. If parent never calls wait(), the zombie persists — wasting a process table entry.

10

Context switch saves & restores the machine state

To switch from A to B: save A's registers into A's PCB → mark A as Ready → load B's registers from B's PCB → mark B as Running → CPU continues from B's saved PC. B never knew it was paused.

Top exam traps to avoid

1. "Ready" ≠ "Running" — Ready means waiting for CPU, not executing.
2. A Blocked process cannot run even if the CPU is free — it's waiting for I/O, not for CPU.
3. Zombie ≠ "crashed" — A zombie finished normally (called exit) but parent hasn't called wait yet.
4. Lazy loading is the modern standard — not eager loading.
5. Mechanism and policy are separate — never say "context switch is a policy" (it's a mechanism).
