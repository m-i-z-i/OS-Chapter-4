§ 4.0 Foundational Concept

What is a Process?

Formal Definitions

Program: A static collection of instructions and data stored on persistent storage (disk). A program is a passive entity — it does not execute independently. It is simply an executable file waiting to be run.


Process: A program in execution. A process is the OS's fundamental abstraction of a running program — an active entity consisting of the program's instructions, its current execution state (register values, program counter), and the resources it is using (memory, open files).

The essential distinction is one of state. A program is an inert file on disk with no dynamic properties. When the OS loads a program into memory and begins executing it, that program instance becomes a process. One program file can give rise to multiple simultaneous processes (e.g., opening two terminal windows runs two independent shell processes from the same executable).

Program — Passive

Static Entity

  • An executable file stored on disk
  • Contains compiled instructions and data
  • Does not consume CPU or memory on its own
  • One copy on disk; can be shared across processes
  • Has no execution state
Process — Active

Dynamic Entity

  • A program currently being executed by the CPU
  • Has a unique execution state (PC, registers, memory)
  • Consumes CPU time and memory at runtime
  • Multiple processes can run from one program
  • Created and destroyed at runtime by the OS
Exam Note

The OS manages processes, not programs. The program is the static blueprint; the process is the active, running instance. This distinction is the foundation of all concepts in this chapter.

Practice Questions
  1. Define a process. How does it differ from a program? In your answer, specify what additional runtime information a process possesses that a program on disk does not. [5 marks]
    Show Answer
    A process is a running program in execution. It differs from a program (which is static code on disk) because it is an active entity. A process possesses runtime information that a program lacks: an execution state (represented by the Program Counter and CPU registers), a private Address Space (containing the stack and heap), and I/O state (open file descriptors).
  2. Explain what is meant by "a process is an abstraction provided by the operating system." [3 marks]
    Show Answer
    The OS provides each running program with the illusion that it has exclusive access to the CPU (via CPU virtualization/time-sharing) and exclusive access to memory (via virtual memory). This constructed environment—the illusion of a dedicated machine—is the abstraction known as a process, isolating programs from each other and the physical hardware limitations.
  3. Can a single program give rise to multiple processes simultaneously? Justify your answer with an example. [2 marks]
    Show Answer
    Yes. A program is just a static file on disk. When the OS loads that file into memory and executes it multiple times concurrently, each execution instance is a distinct process. For example, opening three separate terminal windows runs three independent cmd or shell processes, all originating from the exact same executable program.
§ 4.1 The Crux: Illusion of Many CPUs

CPU Virtualization

Formal Definitions

CPU Virtualization: An OS technique that creates the illusion that each running process has exclusive access to a dedicated physical CPU, while in reality multiple processes share a limited number of physical CPU cores.


Time Sharing (of the CPU): A mechanism in which the CPU is allocated to processes in short intervals called time slices. At the end of each slice, the OS saves the current process's state, selects the next process, and resumes it. By cycling rapidly among processes, the OS creates the appearance of simultaneous execution. Cost: a performance overhead at every context switch.


Space Sharing (of resources like Disk): A resource allocation strategy where the resource is physically partitioned among users permanently. Each partition belongs exclusively to one entity until released — no time-multiplexing occurs.

A physical machine has a fixed number of CPU cores. To allow many more processes to run than there are cores, the OS time-multiplexes the CPU — switching between processes rapidly so each makes progress within a reasonable time frame. The trade-off is a small but real performance penalty at every switch due to saving and restoring registers.

Time Sharing — CPU

Shared Over Time

Run entity A for a time slice, then switch to B. Each process believes it has exclusive CPU access. Cost: performance overhead at each switch (saving/restoring registers and state).

Space Sharing — Disk

Shared Over Space

A resource is divided into partitions. Once a block is assigned to a file, it belongs to that file until deleted. No time switching — different entities occupy different physical regions.

Key Distinction

Time sharing = temporal multiplexing (one resource, divided by time). Space sharing = spatial partitioning (one resource, divided by location). The CPU is time-shared; disk is space-shared.

Practice Questions
  1. Explain how the OS creates the illusion of multiple CPUs using time sharing. What is the performance trade-off involved? [5 marks]
    Show Answer
    The OS multiplexes the CPU by rapidly switching execution among multiple processes in short intervals called time slices. It runs process A, saves its state, runs process B, saves its state, and cycles back so quickly that they appear to run simultaneously. The trade-off is the performance overhead incurred during every context switch, where time is spent saving and restoring registers instead of executing useful user code.
  2. Distinguish between time sharing and space sharing. Provide one concrete example of each in an operating system context. [4 marks]
    Show Answer
    Time sharing divides a single resource into temporal slices, granting entities access one after the other in rapid succession (e.g., the CPU). Space sharing divides a resource into physical partitions, dedicating a piece of the resource exclusively to an entity without time-multiplexing (e.g., Disk block allocation).
  3. Define CPU virtualization and state why it is a fundamental responsibility of a modern operating system. [3 marks]
    Show Answer
    CPU virtualization is the OS technique that provides each process the illusion of having a dedicated physical CPU. It is a fundamental responsibility because it allows a single computer with limited cores to concurrently run hundreds of programs, enabling multitasking while isolating processes from hardware constraints and each other for stability and security.
§ 4.1 OS Design Philosophy

Mechanisms vs. Policies

Formal Definitions

Mechanism: A low-level protocol or procedure that implements a specific OS operation. A mechanism defines how an action is performed but contains no logic about when or why that action should occur.


Policy: A high-level algorithm or decision-making strategy that determines which action the OS should take at any given time. A policy defines which resource to allocate and when, then relies on mechanisms to carry out the decision.

Separating mechanisms from policies is a fundamental principle of OS design. By keeping the low-level implementation of an operation independent from the decision-making logic, the OS gains modularity: policies can be changed or upgraded without altering the underlying mechanism code.

Mechanism — "The HOW"

Low-level Machinery

A protocol for implementation. Contains no decision logic.


Example — Context Switch: The mechanism that saves the current process's CPU registers to memory and loads another process's registers. It performs this when instructed — it does not decide whether a switch should happen.

Policy — "The WHICH"

High-level Intelligence

An algorithm for decision-making. Uses mechanisms to enforce decisions.


Example — Scheduling Policy: Decides which process runs next (round-robin, shortest-job-first, priority, etc.) and triggers the context switch mechanism to enforce that decision.

Design Tip

Modular separation of mechanisms and policies allows policies to change without rewriting low-level mechanisms. OS researchers can experiment with many scheduling algorithms without touching the context switch implementation.

Practice Questions
  1. Distinguish between a mechanism and a policy as used in operating system design. Give a specific example of each. [5 marks]
    Show Answer
    A mechanism is a low-level protocol defining how to perform an operation, without any decision-making logic (e.g., a Context Switch). A policy is a high-level algorithm defining which actions to take and when (e.g., Round-Robin Scheduling). Mechanisms execute actions; policies decide upon actions.
  2. Why is separating mechanisms from policies considered good OS design practice? [3 marks]
    Show Answer
    It enforces modularity. By keeping decision logic separate from implementation details, OS developers can easily update, swap, or experiment with different policies (like new scheduling algorithms) without having to rewrite or risk breaking the complex, low-level mechanism code.
  3. Classify each of the following as a mechanism or a policy and justify: (a) context switch, (b) round-robin scheduling, (c) saving CPU registers to the PCB, (d) priority-based scheduling. [4 marks]
    Show Answer
    (a) Context switch: Mechanism — it's the procedure of switching. (b) Round-robin scheduling: Policy — it decides when to step to the next process. (c) Saving CPU registers to the PCB: Mechanism — it's an implementation detail required to pause a process. (d) Priority-based scheduling: Policy — it decides which process gets the CPU based on rank.
§ 4.1 What a Process Contains

The Machine State

Formal Definitions

Machine State: The complete set of information that uniquely defines the execution context of a process at any instant. It is everything the OS must save and restore accurately to pause and resume a process without the process being aware of the interruption.


Address Space: The private region of memory allocated to a process. It contains the process's instructions (text segment) and data (global variables, heap, stack). No other process may access this memory region.


Program Counter (PC): A CPU register that holds the memory address of the next instruction to be executed by the process. It is the most critical register in the machine state.


Stack Pointer (SP): A CPU register that points to the top of the current call stack, tracking function call frames, local variables, and return addresses.


File Descriptor: An integer handle assigned by the OS to an open I/O resource (file, socket, pipe). A process's I/O state is defined by the set of file descriptors it currently holds open.

Practice Questions
  1. What is the machine state of a process? Identify and explain its three main components. [5 marks]
    Show Answer
    The machine state is the complete set of information defining a process's execution context at any instant. Its three main components are:
    1. Address Space: The isolated memory region containing the program's instructions and data (heap/stack).
    2. Registers: The CPU's execution context, notably the Program Counter (PC) and Stack Pointer (SP).
    3. I/O Information: The list of file descriptors currently open, linking the process to inputs and outputs.
  2. What is the role of the Program Counter (PC) and Stack Pointer (SP) in a process's machine state? Why are they critical to a correct context switch? [4 marks]
    Show Answer
    The Program Counter (PC) points to the memory address of the next instruction to execute. The Stack Pointer (SP) points to the top of the call stack, managing function frames. They are critical to a context switch because if they are not perfectly saved and restored, the process will resume execution at the wrong instruction or with corrupted function tracking, leading to immediate crashing or undefined behavior.
  3. Why must the OS save and restore all components of the machine state during a context switch? What is the consequence if any component is omitted? [3 marks]
    Show Answer
    The OS must save and restore all components to maintain the illusion that the process was never interrupted. If the registers are omitted, execution logic is lost. If I/O state is omitted, file operations will fail. If the memory mapping is omitted, data is corrupted. Omitting any piece breaks the process's abstraction and causes catastrophic failure for the program.
§ 4.2 Process API Overview

The Process API

Formal Definition

Process API: The set of system calls provided by the OS that allow programs and users to create, manage, and destroy processes. Every general-purpose OS must provide at minimum the following five operations.

1

Create fork() / exec()

The OS must be able to create new processes. When a shell receives a command or a user launches an application, the OS is invoked to create a new process. In Unix/Linux this is done via fork() which duplicates the calling process and exec() which replaces the process image with a new program.

2

Destroy kill()

The OS provides an interface to forcefully terminate a process that does not exit voluntarily for example, a process stuck in an infinite loop or one that has become unresponsive. Most processes exit normally via exit(); forced termination is a safety mechanism.

3

Wait wait() / waitpid()

A parent process can pause its own execution until a specific child process completes. This allows orderly parent-child coordination and is the mechanism by which the parent reads the child's exit status and frees the child's PCB (preventing zombie accumulation).

4

Control SIGSTOP / SIGCONT

The OS can suspend a running process and later resume it without terminating it. In Unix, sending SIGSTOP pauses a process; SIGCONT resumes it. This is used by debuggers, shells (Ctrl+Z), and job control systems.

5

Status getpid() / ps / top

The OS exposes interfaces to query the state of a process: its PID, current state (running/sleeping/stopped), CPU usage, memory consumption, and more. Tools like ps and top use these interfaces to list and monitor running processes.

Design Note

The textbook uses FR reference (file reference) for I/O. Every process inherits standard file descriptors (stdin, stdout, stderr) automatically this is part of process creation, not explicit Process API calls.

Practice Questions
  1. List the five operations that every OS Process API must provide. Briefly explain the purpose of each. [5 marks]
    Show Answer
    The five core operations are:
    1. Create: Launch a new process (e.g., fork() and exec()).
    2. Destroy: Forcefully terminate a process (e.g., kill()).
    3. Wait: Pause a parent until a child finishes (e.g., wait()).
    4. Control: Suspend/resume a process (e.g., SIGSTOP/SIGCONT).
    5. Status: Query a process's state and resource usage (e.g., ps or top).
  2. Explain the difference between a process terminating via exit() and being terminated via kill(). Why does the OS need both mechanisms? [3 marks]
    Show Answer
    exit() is a voluntary termination where a process consciously ends its own execution and cleans up gracefully. kill() is an involuntary termination commanded from the outside (by the OS or another user process) destroying the process forcefully. Both are needed because while processes normally finish their work, some may crash, enter infinite loops, or become malicious, requiring a safety override to stop them.
  3. Why is the wait() system call important for process management? What problem does it solve? [3 marks]
    Show Answer
    The wait() system call allows parent processes to pause and synchronize with their child processes. Crucially, it solves the problem of resource leaks by collecting the child's exit status. This allows the OS to delete the child's PCB from the process table, preventing the accumulation of "zombie" processes.
4.3 How Processes Are Born

Process Creation

Formal Definitions

Eager Loading: A process loading strategy in which the OS reads the entire program image (code and data) from disk into memory before execution begins. Simple but slow for large programs.


Lazy Loading (Demand Paging): A process loading strategy in which only the portions of the program image that are actually accessed are loaded from disk, and only when needed. Modern operating systems use lazy loading for faster process startup.

When the OS creates a process, it performs a fixed sequence of steps before executing a single instruction of the user's program:

1

Load the program's code and static data into memory

The OS reads the executable file from disk and loads its instructions (text segment) and initialized data into the process's address space. Modern OSes do this lazily only pages actually accessed are loaded (demand paging), allowing the program to start executing before the entire image is in memory.

2

Allocate and initialize the runtime stack

The OS allocates a fixed-size memory region for the process's call stack and initializes it with the program's command-line arguments (argc and argv[]). The Stack Pointer register is set to point to the top of this stack.

3

Allocate the heap

The OS sets up the process's heap a dynamic memory region that starts small and grows as the program requests more memory via malloc() or new. The OS provides the brk()/sbrk() system call to extend the heap.

4

Set up I/O open standard file descriptors

Before execution begins, the OS opens three default file descriptors: fd 0 stdin (keyboard), fd 1 stdout (terminal output), fd 2 stderr (error output). The program can use these immediately without explicitly opening them.

5

Transfer control to the entry point main()

The OS sets the CPU's Program Counter to the address of the main() function and transfers control. From this point, the process is executing. The OS continues to oversee it through system calls and interrupts, but the process is now running independently.

Practice Questions
  1. Describe the five steps the OS performs to create a new process, in order. [5 marks]
    Show Answer
    1. Load program code and static data from disk into memory (often lazily).
    2. Allocate and initialize the runtime stack (with argv/argc).
    3. Allocate memory for the heap (for dynamic allocation).
    4. Initialize basic I/O (open file descriptors for stdin, stdout, stderr).
    5. Transfer CPU execution control to the program's main() entry point.
  2. Distinguish between eager loading and lazy loading (demand paging) in process creation. Which do modern OSes use and why? [4 marks]
    Show Answer
    Eager loading reads the entire program executable into memory before execution begins. Lazy loading divides the program into pages and only loads the pages precisely when the program attempts to access them. Modern OSes use lazy loading because it vastly improves start-up times, saves memory (by not loading unused features), and allows executing programs larger than available RAM.
  3. Why does the OS pre-open stdin, stdout, and stderr before the program begins executing? [2 marks]
    Show Answer
    Pre-opening these file descriptors provides a standardized interface for basic IO right out of the box. The program does not have to interface manually with the keyboard or monitor drivers; it can immediately use printf() or cin because the pipes (fd 0, 1, 2) are guaranteed to be connected by the OS from the very first instruction.
4.4 Process Lifecycle

Process States

Formal Definitions Five Process States

New (Start): The process is being created. The OS is allocating and initializing the PCB, loading the program, and setting up memory. The process has not yet been admitted to the ready queue.


Ready: The process is fully initialized and waiting in the ready queue to be assigned to a CPU. It is capable of executing immediately if a CPU becomes available.


Running: The process is currently executing on the CPU the CPU's Program Counter points to this process's instructions. Only one process per CPU core can be in the Running state at any moment.


Waiting (Blocked): The process cannot proceed because it is waiting for an external event to complete typically an I/O operation (disk read, network packet, keyboard input). It cannot use the CPU even if one is free.


Terminated: The process has finished execution (called exit()) or was forcibly killed. Its resources are being freed by the OS. The process may briefly enter the Zombie state before full cleanup.

A process executes instructions ONLY in the RUNNING state. Ready = capable of running but waiting for the CPU. Waiting/Blocked = cannot run even if the CPU is free it is waiting for an external event. Never confuse Ready with Running.

Practice Questions
  1. Draw and explain the five-state process state diagram. For each state, describe what condition the process is in. [5 marks]
    Show Answer
    (Draw diagram with 5 nodes: Start, Ready, Running, Waiting, Terminated, with arrows mapping transitions.)
    New/Start: Being created, not yet admitted to ready queue. Ready: In queue, capable of executing immediately, waiting for CPU. Running: Currently executing instructions on the physical CPU. Waiting/Blocked: Suspended, unable to run until an external event (like I/O) finishes. Terminated: Finished execution, resources being freed.
  2. List all possible state transitions in the process lifecycle and state the trigger for each transition. [5 marks]
    Show Answer
    1. New → Ready (Admit): Process finishes initialization.
    2. Ready → Running (Dispatch): Scheduler assigns the CPU to to the process.
    3. Running → Ready (Pause/Preempt): Process's time slice expires or a higher priority task preempts it.
    4. Running → Waiting (Event Wait): Process initiates I/O or requests a lock.
    5. Waiting → Ready (Event Completion): Missing I/O data arrives.
    6. Running → Terminated (Release): Process calls exit() or is killed.
  3. A process is in the Waiting/Blocked state. Explain why it cannot run even if the CPU is completely idle. [3 marks]
    Show Answer
    A process enters the Blocked state when it requires data from an external, slower device (like a hard drive or network card) to proceed. It lacks the necessary data to execute its next instruction. Therefore, even if an idle CPU were assigned to it, the process would simply stall doing nothing. It must remain blocked until the hardware interrupts the OS to announce the data is ready.
  4. Distinguish between the Ready state and the Running state. [2 marks]
    Show Answer
    In the Running state, the process is currently occupying the CPU hardware and actively executing machine instructions. In the Ready state, the process wants to execute and has all data it needs, but it is waiting in a queue because the CPU is currently occupied by another process.

4.5 Data Structures

The Process Control Block (PCB) and Context Switch

Formal Definitions

Process Control Block (PCB): A kernel data structure that stores all information the OS needs to manage a single process. There is exactly one PCB per process. The OS maintains a list (process table) of all PCBs for all existing processes.


Context Switch: The mechanism by which the OS saves the machine state (registers) of the currently running process into its PCB, and restores the machine state of another process from that process's PCB, effectively switching CPU execution from one process to another. It is a mechanism, not a policy.


Zombie State: A process that has completed execution (called exit()) but whose PCB has not yet been removed because the parent process has not called wait() to collect its exit status. A zombie holds no memory or CPU resources only a PCB entry with its PID and exit code.

The OS tracks every process in a list using a PCB. The table below shows the xv6 teaching OS implementation of the PCB:

proc.h Process Control Block (xv6)
struct proc {
    /* State this process is in */
    enum proc_state  state;    // RUNNING, READY, BLOCKED, ZOMBIE, etc.

    /* Saved CPU registers  stored here during a context switch */
    struct context   context;  // Contains PC, SP, and all general-purpose registers

    /* Memory */
    char             *mem;     // Pointer to start of process address space in RAM

    /* Process relationships */
    struct proc     *parent;  // Pointer to parent process (creator)

    /* Identifier */
    int              pid;      // Process ID  unique integer assigned by the OS
};
state

Current State

The process's current state enum: RUNNING, READY, BLOCKED, or ZOMBIE. Updated by the OS at every state transition.

context

Saved Registers

On every context switch, all CPU registers (including PC and SP) are saved here. When this process runs again, they are restored exactly this is how seamless resumption is achieved.

mem / pid / parent

Identity & Memory

mem: pointer to address space. pid: unique process ID (init = PID 1). parent: pointer to the process that created this one used for wait() and zombie cleanup.

Context Switch Formal Procedure

1

Save current process (A) state

The OS writes all CPU registers (PC, SP, general registers) from the CPU into Process A's PCB.context. Process A's state is updated to READY (or BLOCKED).

2

Select next process (B)

The scheduling policy selects which process to run next (this is the policy's decision not the context switch mechanism's concern).

3

Restore selected process (B) state

The OS loads all registers from Process B's PCB.context back into the CPU. Process B's state is updated to RUNNING. The CPU's Program Counter now points to wherever B left off B resumes seamlessly.

The Zombie State

1

Child process calls exit(status_code)

Execution ends. The OS frees the process's address space (RAM), open file descriptors, and other resources. However, the PCB cannot be deleted yet it holds the exit status code that the parent may need.

2

Process enters ZOMBIE state

The process is "mostly dead" it has no memory, cannot execute, yet its PCB entry (with PID and exit code) still exists in the process table. It is visible in ps output.

3

Parent calls wait() PCB is deleted

When the parent calls wait(), the OS delivers the child's exit status to the parent and completely removes the child's PCB. The zombie is fully cleaned up.

!

Problem: Parent never calls wait()

If the parent does not call wait(), the zombie's PCB persists until the parent itself exits. If thousands of zombies accumulate, the process table fills, preventing the creation of new processes a resource exhaustion bug or attack.

Exam Definition to Memorize

Zombie state: A process that has called exit() and finished executing, but whose PCB entry has not yet been removed because its parent process has not called wait(). The zombie holds no memory or file descriptors only a PCB entry with PID and exit status.

Practice Questions
  1. What is a Process Control Block (PCB)? List the information it stores and explain the purpose of each field. [5 marks]
    Show Answer
    A PCB is a kernel data structure that tracks all information for a single process. Key traits include:
    1. State: Current process status (Ready, Running, Blocked, etc.) used by the scheduler.
    2. Context: The saved CPU registers (PC, SP, etc.) used to pause and resume the process.
    3. Memory Pointers: Locations mapping the process address space (code/data/heap).
    4. Parent: A pointer to the creating process for signalling completion via wait().
    5. PID: The numerical identifier for OS management.
  2. Define a context switch. Describe the exact steps the OS performs during a context switch from Process A to Process B. [5 marks]
    Show Answer
    A context switch swaps the CPU from one running process to another without destroying either. Steps:
    1. Stop executing Process A.
    2. Copy Process A's current CPU registers into A's PCB (saving state).
    3. Change A's state to Ready or Blocked.
    4. Run scheduling policy to select Process B.
    5. Change B's state to Running.
    6. Load registers from B's PCB onto the CPU hardware.
    7. Execution seamlessly resumes at B's restored Program Counter.
  3. What is a zombie process? Explain why it exists and what happens if the parent process never calls wait(). [4 marks]
    Show Answer
    A zombie process is a process that has finished execution (called exit()) but whose PCB still exists in the OS process table. It exists to store the exit status code. The zombie only dies when the parent process calls wait() to read those exit details. If the parent forgets to ever call it, the PCB stays in RAM forever. If thousands accumulate, it exhausts system PID allocation and limits creating new processes.
  4. Is the context switch a mechanism or a policy? Justify your answer. [2 marks]
    Show Answer
    It is fundamentally a mechanism. The context switch is a procedural implementation that knows exactly how to save and restore registers. It performs this low-level swap without caring which process is selected or why it was stopped. The logic to decide who to switch to belongs to the scheduling policy.

Chapter 4 Key Terms Glossary & Exam Guide

The following glossary covers every term your teacher identified as likely exam topics. Mark distribution patterns: 5+5 or 3+3+4.

Formal Glossary Chapter 4 Key Terms
TermFormal Definition
Process A program in execution. An active entity comprising the program's instructions, its current execution state (registers, PC), and resources in use (memory, I/O).
Program A static collection of instructions and data stored on disk. A passive entity with no execution state of its own.
CPU Virtualization The OS technique of creating the illusion that each process has its own dedicated CPU core by rapidly time-multiplexing the physical CPU among all running processes.
Time Sharing A CPU resource allocation strategy in which the CPU is divided into small time slices and rapidly switched among processes. Cost: a performance hit at each switch.
Mechanism A low-level OS protocol that implements how an action is performed. Contains no decision logic. Example: context switch.
Policy A high-level OS algorithm that decides which action to take and when. Uses mechanisms to enforce decisions. Example: scheduling policy.
Machine State The complete information that defines a process's execution context: (1) Address Space, (2) CPU Registers (PC, SP), (3) I/O Information (open file descriptors).
Context Switch The mechanism of saving the current process's registers into its PCB and loading another process's registers from its PCB, switching CPU execution between processes.
PCB Process Control Block. The OS kernel data structure tracking one process. Fields: state, context (saved registers), mem (address space pointer), parent, pid.
Process States Five states: New (being created), Ready (waiting for CPU), Running (on CPU now), Waiting/Blocked (waiting for I/O), Terminated (finished).
Zombie State A process that has called exit() and finished executing but whose PCB persists because the parent has not called wait() to collect its exit status.
Demand Paging Lazy loading strategy: program pages are loaded from disk only when they are first accessed, rather than all at startup (eager loading). Used by all modern OSes.
Common Exam Traps Do Not Make These Mistakes

1. "Ready" "Running" Ready means waiting for the CPU, not executing on it.
2. A Blocked process cannot run even if the CPU is completely free it is waiting for an I/O event, not for the CPU.
3. Zombie crashed process A zombie is a process that exited normally but hasn't been cleaned up by its parent yet.
4. Context switch is a mechanism never say it is a policy. The scheduling algorithm is the policy.
5. Lazy loading (demand paging) is the modern standard not eager loading.
6. One program can spawn many processes they are not the same thing.
